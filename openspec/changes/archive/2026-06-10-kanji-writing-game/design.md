## Context

新規の Next.js（App Router、Next 16 / React 19）プロジェクト上に、漢字書き取りゲームのプロトタイプを構築する。現状は雛形ページ（`src/app/page.tsx`）のみ。手書き（Canvas）と音声（`SpeechSynthesis`）はいずれもブラウザ専用 API のため、ゲーム本体はクライアントコンポーネントで実装する。サーバーやデータベースは使わず、漢字データはアプリ内に静的に持つ。

注意: このリポジトリの Next.js は学習データと挙動が異なる可能性がある（`AGENTS.md`）。実装フェーズでは `node_modules/next/dist/docs/` の該当ガイドを参照してから書く。

## Goals / Non-Goals

**Goals:**
- 「読みの提示 → 虫食い例文表示 → 手書き → 画像類似度で採点 → ポイント加算 → 音声読み上げ」の一連が数問で動くこと。
- 外部 API・課金・サーバー処理なしで完結すること（手書きは Canvas、音声は Web Speech API）。
- 漢字データを後から拡張しやすい構造にすること。

**Non-Goals:**
- 正式な手書き文字認識（OCR/ML）による厳密な正誤判定は行わない（類似率ベース）。
- 小1漢字80字の完全網羅、学習進捗の永続化、ログイン、ランキングは対象外。
- 筆順の判定は対象外。

## Decisions

### 1. クライアント完結のSPA的構成
ゲーム画面は `"use client"` のクライアントコンポーネントとして実装し、状態（現在の問題番号・累計ポイント・採点結果）は React state で管理する。
- 理由: Canvas と `SpeechSynthesis` がブラウザ専用で、サーバー往復が不要なため。
- 代替案: API Route で採点 → 不要なレイテンシと複雑性。却下。

### 2. 手書きは HTML Canvas（Pointer Events）
`<canvas>` に Pointer Events（`pointerdown/move/up`）で描画。タッチ／ペン入力を主対象とし、PC のマウスも同じ Pointer Events で扱う（デバイス分岐は不要）。タッチ時のスクロール抑止に `touch-action: none` を指定。線は黒・一定の太さ・丸端で描く。
- 採点用画像は `canvas.getContext('2d').getImageData()` から取得する。
- 代替案: 外部手書きライブラリ → MVP には過剰。標準 API で足りる。

### 3. 採点は画像類似度（正規化 + ピクセル比較）
お手本画像は、対象漢字を標準フォントでオフスクリーン Canvas に描画して生成する。手書き画像・お手本画像の双方を次の手順で比較する。
1. それぞれ2値化（インクのある画素を前景とする）。
2. 前景のバウンディングボックスを求め、共通サイズ（例: 64×64）にリサイズして位置・大きさを正規化する。
3. 画素一致度を算出する。指標は IoU（前景集合の重なり / 和集合）を基本とし、必要に応じて軽いブラー後の一致率で平滑化する。
4. 一致度を 0〜100% にスケールして類似率とする。
- 理由: 手書きはキャンバス内の位置・サイズがばらつくため、正規化なしの素朴なピクセル差は無意味になる。IoU は位置正規化後の形状一致を素直に表す。
- 代替案: ストローク/軌跡比較（DTW）→ お手本軌跡データの整備が必要でMVPに不向き。手書き認識API → 外部依存・コスト増で却下。

### 4. 採点フローはお手本重ね表示 → %スコア配点
採点時はまずユーザーの手書きにお手本漢字を重ねて表示し、自分の字との違いを目視できるようにする。そのうえで類似率（%）を提示し、ポイントを配点する。類似率は合否判定ではなく応援スコアとして扱う。
- 獲得ポイント = 類似率に対する単調増加関数（例: `round(類似率 × 最大ポイント / 100)`、もしくは閾値による段階配点）。類似率0%は0点。
- お手本重ね表示があることで、画像類似度の判定精度が多少甘くても、ユーザー自身が正しい字形に気づける（採点精度の弱さを学習体験で補う）。
- MVP では線形配点で十分。演出（しきい値で「すごい！」等）は実装時の調整事項。

### 5. 音声は Web Speech API
`window.speechSynthesis.speak(new SpeechSynthesisUtterance(text))` を `lang='ja-JP'` で使用。読み上げ対象は虫食い前の元例文。未対応（`!('speechSynthesis' in window)`）なら再生ボタンを無効化し、他機能は継続。
- 代替案: クラウドTTS／録音 → コスト・運用・拡張性の観点でMVP外（proposal の判断どおり）。

### 6. データ構造
```ts
type KanjiQuestion = {
  kanji: string;        // 対象漢字（1文字）
  reading: string;      // 読み（例: "やま"）
  sentence: string;     // 元例文（対象漢字を含む完全文）
};
```
虫食い表示は `sentence` 中の `kanji` を `□` に `replaceAll` して生成し、対象位置を別途持たない（位置は文字列から導出）。音声読み上げは `sentence` をそのまま使う。お手本画像も `kanji` から生成するため、データは最小限で済む。
- 配当の正確さのため、データには小1配当漢字のみを入れる。

## Risks / Trade-offs

- [画像類似度は字形のズレ（へん／つくりの崩れ、画数違い）に寛容で、別字でも高スコアになりうる] → 採点後に必ずお手本を重ねて表示し、ユーザー自身が違いに気づける設計で補う（Decision 4）。MVP は「書く練習」を主目的とし厳密判定は非目標と明示。正規化サイズ・指標は実装時に数問でチューニング。
- [`SpeechSynthesis` の日本語音声・読み品質はブラウザ/OS依存] → 非対応時フォールバックを仕様化済み。読みは `reading` フィールドで担保し、音声は補助とする。
- [Canvas のタッチ操作がスクロールと競合] → `touch-action: none` と `preventDefault` で抑止。
- [Next.js のバージョン差異による API 差異] → 実装前に `node_modules/next/dist/docs/` を参照（AGENTS.md 準拠）。

## Open Questions

- 配点カーブは線形か段階（しきい値）か。→ 実装時に数問で体感調整。
- 正規化サイズ（64×64 等）と類似指標（IoU 単体か平滑化併用か）の最終値。→ プロトタイプで実測して決定。
- 1問あたりの再挑戦可否（採点後にやり直せるか）。→ MVP は1回採点→次へを基本とし、必要なら後追加。
